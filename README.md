# Song Recommendation System

Content-based song recommender built on the Spotify Tracks Dataset (114,000 tracks, 114 genres). Recommends similar songs using cosine similarity over audio features and genre, with an MMR re-ranking step for variety. No user/ratings data exists in this dataset, so collaborative filtering isn't used here.

## Setup

```bash
pip install -r requirements.txt
```

## 1. Run the ML pipeline

Builds the model and writes everything the API needs into `models/`:

```bash
python main.py
```

Output: `models/feature_matrix.pkl`, `scaler.pkl`, `feature_columns.json`, `songs_lookup.pkl`, `model_metadata.json`.

You only need to re-run this if you change the dataset or the feature engineering code.

## 2. Run the API

```bash
uvicorn app.main:app --reload
```

Docs at `http://localhost:8000/docs`. Endpoints:

| Endpoint | Description |
|---|---|
| `GET /health` | model status + eval metrics |
| `GET /search?q=love&limit=20` | search by track name / artist |
| `GET /recommend/{track_id}?n=10&method=mmr` | get recommendations (`method`: `mmr` or `baseline`) |
| `GET /genres` | list all genres in the catalog |

## 3. Run the frontend

`frontend/index.html` is a single static file вЂ” just open it in a browser. It calls the API at `http://localhost:8000` by default; edit `API_BASE` at the top of the `<script>` tag if the backend runs elsewhere.

## Tests

```bash
pip install -r requirements-dev.txt
pytest
```

24 tests covering `src/preprocessing.py`, `src/features.py`, and `src/recommender.py` (baseline recommender, MMR, and search). Runs against a 3,000-row sample of the real dataset, not synthetic data, so the suite exercises the actual pipeline while staying fast (a few seconds).

## Docker

Optional вЂ” not required to run the project locally.

```bash
python main.py                 # generate models/ first, on the host

docker run -p 8000:8000 song-rec
```

## Notes

- No collaborative filtering: this dataset has no user/ratings data, only track-level audio features, so a content-based approach is the correct fit.
- `mmr` is the default recommender вЂ” better catalog coverage and intra-list diversity than plain top-N, at a modest genre-precision cost. See `model_metadata.json` в†’ `eval_summary` after running the pipeline, or the notebook's Evaluation section for the full writeup.
- Train/test split doesn't apply here вЂ” there's no label being predicted. Evaluation instead uses held-out query seeds and proxy metrics (genre precision, diversity, coverage), the standard approach for recommenders without ground-truth ratings.
